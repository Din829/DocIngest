# 竞品定位与战场边界

> 这份文档给做产品方向决策的人看（维护者 / 未来的自己 / 接手 Agent）：DocIngest 在
> 文档前处理这条赛道上，真实的位置在哪、用户为什么选它、哪些仗该打哪些该让。
>
> **写作原则**：
> - **讲地位和用户场景，不堆技术点。** 单个技术（某个渲染器、某个 triage）突出了没意义；
>   重要的是「谁、在什么场景下、为什么选你而不是别人」。
> - **客观，不吹不黑。** 承认打不过的地方，也不把对手说成废物。凡结论标证据来源
>   （官方文档 / 竞品源码 / 本仓库实测 / 第三方横评）。
> - **事实会过期。** 底部有「复核触发条件」，竞品出大版本时回来重核；事实优先于旧结论。
>
> 调研时间：2026-06。云三家以官方 learn/docs 为准；本地竞品含源码精读与第三方横评。

---

## 0. 一句话定位

**DocIngest 是「不上云、或上不起云的那批人，把日常文档（PDF / Excel / Office 为主）
变成能搜索的知识库」的本地工具。**

不是云厂商方案的替代品（云上打不赢），也不是又一个 PDF 转换器（本地一堆了）。它的位置是
两者之间一块没人好好占的空地：**本地、低门槛、主流文档处理扎实、直接出 RAG 能用的产物。**

> **核心是 PDF / Excel / Office 这些日常文档**——这是绝大多数用户真正要处理的，也是 DocIngest
> 该打磨到最好的地方。音视频 / URL 是**附加能力**（顺带也能做、本地竞品确实没有），但**不是
> 主轴、不是大多数用户来的理由**——别把定位押在它上面。下文凡涉及音视频，都按「附加」对待。

## 1. 赛道里有三类玩家，DocIngest 是第三类

| 类 | 代表 | 用户是谁 | 形态 |
|---|---|---|---|
| **云端全家桶** | Azure AI Search / AWS Bedrock KB / GCP Vertex | 已经在某朵云上、要端到端 RAG 的企业 | 托管云服务，解析→切块→向量化→检索全包 |
| **本地转换器** | Docling / MinerU / Marker / markitdown / PyMuPDF4LLM | 要把文件转成 md 的开发者 | pip 装的库，多数做到「文件→md」就停 |
| **本地管线** | **DocIngest** | 要本地建知识库、输入杂、不想碰云的人 | pip 装，但解析+理解+切块+索引一条龙 |

**关键区别不是「谁功能多」，是「服务谁」**：
- 云端那类，用户是「已经买了云、要省事」的。
- 本地转换器那类，用户是「我只要把这个 PDF 变 md」的。
- DocIngest，用户是「我要把一堆日常文档（PDF、合同、Excel、Word，偶尔也有录音/视频）变成
  一个能搜的知识库，而且我不想/不能把数据交给云」的。

三类用户**重叠不多**，所以下面分开谈打不打得过。

## 2. 对云厂商：云上让出去，因为那本来就不是 DocIngest 的用户

### 2.1 云三家在做同一件事，机制和 DocIngest 同源

2025-2026 三家云都把「解析+切块+图理解+检索」绑进自家服务，底层都走 Vision 多模态 LLM +
layout 模型，和 DocIngest 同源，**功能上「差不多」是真的**：

| | Azure | AWS Bedrock KB | GCP |
|---|---|---|---|
| 解析+切块 | Content Understanding（2026-04 GA） | Bedrock Data Automation | Layout Parser v1.6（2026-01） |
| 图理解 | GenAI Prompt skill（Vision） | Claude / Nova 多模态 | Gemini layout parser |
| 音视频 | ❌ 要另接服务 | ✅ Nova（原生支持） | ❌ |
| 检索 | Knowledge bases | KB 原生 | Vertex AI Search |

（来源：各家官方 learn/docs 站，2026-06）

### 2.2 云上打不赢，但这不是问题——那不是 DocIngest 的用户

正面打必输，原因是结构性的：用户数据已在云上、要的是端到端+SLA+合规背书、检索是云的主场而
DocIngest 不做检索。**这些都补不上，也不该补。**

**但要看清：会用 Azure AI Search 的人，本来就不会用 DocIngest。** 他们已经决定上云、有预算、
有团队配 skillset。DocIngest 去抢这批人是去人家主场送死。**云上这块主动让出，不是认输，是
不打没用户的仗。**

### 2.3 云厂商够不着的那批用户，才是 DocIngest 的

云方案有个共同特征：**门槛高、绑生态、数据进它边界。** 这把一大批用户挡在门外——而这批正是
DocIngest 的：

- **嫌开通麻烦的**：跑通一个 Azure ingest，要开订阅→建 Search 服务→建 Foundry 资源→部署
  模型+申请配额→配权限→写一堆 JSON→数据进 Blob。对比 DocIngest：装上、填个 key、一行命令。
- **数据不能出网的**：日本很多传统企业 / 金融 / 政府对数据出境极敏感。云方案数据必须进云边界。
- **上不起云的**：云唯一的「真本地离线」方案（Azure DI disconnected 容器）要求被微软认证为
  「战略客户」+ 预付承诺消费 + 8 核 16GB 起 + 走审批——中小团队根本够不着。
  （来源：learn.microsoft.com，DI disconnected containers）

> **为什么云厂商不做低门槛本地版**：技术上，图理解/强解析是云端 GPU 服务，搬本地就变重变贵；
> 商业上，让免费本地工具变强 = 砸自己的云收入。所以云厂商**不是不会做，是商业上不做**。
> 这块空地，结构性地留给了不卖云的玩家。

## 3. 对本地竞品：这才是真正的战场，按「谁该用谁」看

### 3.1 本地工具：每个都有清晰的用户画像（含第三方横评定论）

第三方横评（themenonlab 2026 PDF-to-md，独立来源）给每个工具的「best for」：

| 工具 | 横评定论：best for 谁 | 不做什么 |
|---|---|---|
| **Marker** | 通用转换、多格式管线、想要一个工具搞定的团队 | 无音视频；切块支持有限 |
| **Docling** | 企业文档处理、已在 LangChain/LlamaIndex 生态的团队 | 视频无；CPU 慢；切块靠外部框架 |
| **MinerU** | CJK 文档、复杂学术论文、有 GPU 的团队 | 要 GPU；仅文档类；无音视频 |
| **pdf-craft** | 数字化纸质书、图书馆/档案项目 | 仅 PDF/书；社区小 |
| **PyMuPDF4LLM** | 原生 PDF、快速提取、纯 CPU 环境 | 无 OCR，扫描件死；纯文本 |
| **markitdown** | office→md 快速转换、要广格式覆盖的轻量场景 | 见 §3.3 |

> **关键：横评里压根没提 DocIngest。** 所以这张表不是「第三方在夸 DocIngest」，而是反过来——
> **它证明了空白**：横评里**没有一个工具的「best for」是「要直接出 RAG 知识库 / 日本那种画在
> 格子里的 Excel / 一个目录杂输入一把抓」**（音视频更是全员空白，但那是附加项）。这几块用户，
> 本地生态目前没人对口。

### 3.2 DocIngest 对口的是哪批用户（本地竞品没覆盖的）

不比「谁解析更准」（那个比不赢，见 §3.4），比「谁服务这批人」。**先看核心场景（大多数用户来的
理由），再看附加能力（顺带的加分项）**：

**核心（PDF / Excel / Office 为主）——这是主战场：**

| 这批用户要的 | 本地竞品现状 | DocIngest |
|---|---|---|
| **PDF / Office** 解析后**直接出能喂 RAG 的切块产物**，不想自己再切 | 多数到 md 就停，切块靠外部框架 | 有（chunks.jsonl + 元数据，开箱即用） |
| 文档是**日本那种画在格子里的 Excel（方眼紙）/ 仕様書** | 横评原话：没人专门处理方眼紙 | 有（专门处理） |
| 输入杂：一个目录里 **PDF / Excel / Word / ZIP** 混着，想一把抓 | 多数单格式或几格式 | 有 |
| 要**数据不出本机 + 换不同 AI 模型** | 部分支持 | 有 |

**附加（不是大多数人来的理由，但本地竞品确实没有，算加分）：**

| 附加能力 | 本地竞品现状 | DocIngest |
|---|---|---|
| 偶尔有**录音 / 会议视频 / 视频链接**也要进知识库 | 横评里只有 Docling 沾音频，视频全员空白 | 顺带也能做（音频 / 视频 / URL） |

**一句话**：本地竞品大多是「把一种文件转好 md」的专才；DocIngest 是「把一个目录里杂七杂八的
**日常文档**变成一个能搜的知识库」的通才管线。**对口的是「要建知识库、文档多样、要本地」的人，
不是「我只想转个 PDF 求最准」的人。** 音视频是这套通才能力的自然延伸，不是它存在的理由。

### 3.3 markitdown（微软的、名头大，最该看清楚）

读了它的源码（本仓库 `参考项目/markitdown/`），客观结论是：**它是一个格式覆盖很广的轻量
转换器，但本地能力浅，真要「强」就转接 Azure 云。** 不是废物，是定位不同。

事实（源码实证，不带情绪）：
- **覆盖广**：20+ converter（PDF/Office/图片/音频/youtube/zip/epub/outlook/rss/wikipedia…），
  内容探测用 magika。格式数量上不输谁。
- **但每个浅**：PDF 是纯文本层 + 坐标凑表（`_pdf_converter.py`），**不看页面图像，扫描件无内容**；
  xlsx 是 `pandas → to_html`（`_xlsx_converter.py` 13 行），方眼紙合并单元格会糊成一坨。
- **图理解 / 强解析是「转接头」不是本地能力**：图描述把图发给 OpenAI（`_llm_caption.py`，默认关、
  要你自己传 client）；要强解析则装 `az-doc-intel` / `az-content-understanding` 两个 opt-in extra
  **转接 Azure 云**（pyproject 坐实）。
- **音视频是「拿现成的」**：音频用 Google 免费网页 ASR；YouTube 只拉**官方字幕**，没字幕就没内容，
  不下载不转录。

> **它和 DocIngest 不在比同一件事**：markitdown 的角色是「轻量转换 + 把人引向 Azure 云做重活」；
> DocIngest 的角色是「本地就把重活（含音视频、版面理解、切块）做完」。markitdown 越流行，越说明
> 「本地+输入杂+不绑云」这批用户的需求是真的——但微软的免费工具被商业模式焊死，不会去做本地重活
> （会砸 Azure 收入），这个位置只能由不卖云的玩家来占。

### 3.4 诚实承认：DocIngest 不是样样赢

- **纯 PDF→md 解析精度**：DocIngest 解析底座用的就是 Docling，所以这块 ≈ Docling，**不超它**；
  MinerU 在 CJK / 复杂版面上比 DocIngest 强。**「解析最准」不是 DocIngest 的卖点，别去碰。**
- **GPU 极致速度**：Marker / MinerU 上 GPU 更快。
- **DocIngest 的价值不在「某一格式转得最好」，在「服务一批别人没服务的用户」**——文档多样（PDF/
  Excel/Office 为主）、要本地、要直接出知识库（音视频是附带延伸）。定位是「日常文档处理最扎实、
  覆盖最全的本地管线」，不是「解析最强的转换器」。

### 3.5 真正的护城河：日式 Excel 样本（数据飞轮，竞品复制不了）

前面讲的能力（切块、方眼紙处理、本地）竞品迟早能补。**唯一难复制的是「样本来源」——
DocIngest 能持续接触到大量真实日企的各种各样 Excel 式样书，这是开源竞品拿不到的。**

为什么这是最硬的优势：

- **日式 Excel 是「见得越多、做得越好」的活**：日企的設計書/仕様書把 Excel 当画布画（方眼紙）、
  合并单元格、跨页表、画面レイアウト、テーブル定義、帳票、ER 図、業務フロー…**形态千奇百怪，
  没有统一标准**。处理它不能靠硬编码规则，只能靠**见过足够多真实样本**才能抽象出稳的结构规则
  （这正是 `SOURCE_MD_CONTRACT.md` / `CHUNKER_FLEXIBILITY_ISSUES.md` 的核心思想：靠结构不靠关键词，
  而结构规则要真实样本喂出来）。
- **竞品拿不到这些样本**：Docling/MinerU/markitdown 是开源团队，只能对着公开数据集调。**真实日企
  内部的奇形怪状 Excel，他们见不到**，也就没动机、没数据为方眼紙调优。
- **DocIngest 有真实样本流**：业务上能持续接触日企文档（咨询资料预处理等）。本仓库 `test_docs/`
  已有大量真实样本——和歌山市 / 宮城県 / 総務省（政府）、bizroute / moonmile / foox（企业）的
  画面レイアウト定義書、テーブル定義書、帳票レイアウト、機能一覧、見積書…，且同种文档常有多个
  真实变体。
- **这是数据飞轮**：样本越多 → 抽象出的结构规则越泛用 → 产物越好 → 越能服务日企 → 越多样本。
  **越滚越深，竞品想追要先解决「拿不到样本」这个上游问题，绕不过去。**

> **定位含义**：日本场景不是「我们顺便也支持日语」，是「我们手里有别人拿不到的真实日式 Excel
> 样本，能把这块做到别人做不到的程度」。这是 DocIngest **最该长期投入、也最难被复制**的一块。

## 4. 结论：DocIngest 的位置

> **服务对象**：要在本地建知识库、文档多样（PDF / Excel / Office 为主）、数据不想/不能上云的人——
> 中小团队、对数据敏感的日企、不想碰云门槛的开发者。
>
> **为什么选它**：本地竞品大多只转一种文件、到 md 就停、方眼紙处理糙；云方案门槛高又绑生态。
> DocIngest 是这中间唯一「低门槛 + 主流文档扎实 + 直接出 RAG 产物 + 不绑云」的。（音视频 / URL
> 是附加项，锦上添花，不是主卖点。）
>
> **最难被复制的一块**：日式 Excel 式样书处理——因为能持续接触到真实日企样本，这是开源竞品
> 拿不到的数据护城河（见 §3.5）。**这是该长期重投的方向。**
>
> **不抢谁的用户**：不抢「已经上云要端到端」的（让给 Azure/AWS/GCP），不抢「只想转个 PDF
> 求最准」的（让给 MinerU/Docling）。

## 5. 这对定向开发意味着什么

把「服务谁」翻译成开发优先级。**判断一个功能做不做，问「它是在更好地服务我们那批用户
（本地、输入杂、要知识库、不绑云），还是在往别人的用户群里冲？」**

### 5.1 该投入（按优先级，更好服务「我们的用户」）

**核心（大多数用户的日常，优先级最高）：**
- **PDF / Excel / Office 主流文档处理打磨到最好**：解析质量、表格还原、版面理解——这是绝大多数
  用户真正天天用的，DocIngest 的根基在这，资源优先往这投。
- **日式 Excel 式样书——最该长期重投的护城河（见 §3.5）**：方眼紙、画面レイアウト定義書、
  テーブル定義書、帳票、見積書…各种真实日企形态。**做法是数据飞轮**：拿真实样本喂出泛用的
  结构规则（靠结构不靠关键词），新见到一种奇形怪状的就补一个 fixture、抽象进规则，越滚越稳。
  **工程纪律：小侵入性地加**——每次只针对真实样本补规则，不为单个 case 硬编码、不推翻已有逻辑、
  改完核对不破坏其他格式（呼应 `SOURCE_MD_CONTRACT.md` 的契约）。这块别人没样本，追不上。
- **出口更好用**：切块产物的质量、与下游（向量库 / LangChain）无缝对接——对口「要建知识库」的人。
- **降低使用门槛**：装得更简单、跑得更省心、GUI / 一键 exe——「为什么能选你」的根基，对口「嫌云麻烦」的人。
- **数据可控 + 可审计**：本地不出网、能换模型、产出可信（质量报告）——对口「数据敏感 / 合规」的人。

**附加（维持可用即可，别当主力投入）：**
- **音视频 / URL**：保留为附加能力（本地竞品确实没有，是加分项），维持能用、跟上 provider 即可；
  **不是投入重心**——它不是大多数用户来的理由，把它做成「锦上添花」而非「核心卖点」。

### 5.2 该克制（往别人用户群冲，没意义）

- 不做云端检索 / 向量索引主流程——那是云的用户，DocIngest 只做前处理。
- 不和 MinerU 卷「纯解析精度」——底座是 Docling，卷不赢，也不是我们用户的核心诉求。
- 不做「Azure 替代品」叙事——会把定位往打不赢的战场拖。
- 不追 GPU 极致速度——不是我们用户的痛点。

### 5.3 一条立项检验线

新功能问一句：**「这是在帮『本地建知识库、输入杂、不绑云』的人，还是在帮别的人？」**
帮我们的人 → 做；帮别人的人（云用户 / 只求解析最准的人）→ 停。

## 6. 复核触发条件（事实会变，定期回来核对）

本定位基于 2026-06 的事实。下列情况发生时，回来重核对应章节，**事实优先于旧结论**：

- **云厂商出「真低门槛本地版」**（pip 级、无审批、数据不出网）→ §2.3「云够不着」可能失效。
- **本地竞品补齐音视频 / 方眼紙 / 切块产物**（如 Docling 加视频、MinerU 加方眼紙）→ §3「空位」收窄。
- **markitdown 本地能力实质增强**（加本地 Vision/OCR/切块，而非继续转接云）→ §3.3 更新
  （商业上不太可能，见 §2.3）。
- **DocIngest 换掉 Docling 解析底座** → §3.4「解析精度 ≈ Docling」需更新。

复核方式：拿本仓库 `test_docs/` 真实样本（含方眼紙 Excel、仕様書、音视频）跑对照，眼见为实。
